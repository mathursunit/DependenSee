What Carrier DependenSee does

Carrier DependenSee maps a Windows machine so you can plan its move to the cloud with confidence. It answers three questions:

  • What services are installed and running? Every registered Windows service, with state, start mode, and owning process.
  • What is this machine listening for? Every program with an open network port.
  • Who does it talk to over time? Every inbound and outbound connection, sampled continuously and attributed to the program responsible — the raw material for the firewall rules you'll need on the other side.

CollectorA background Windows Service that records data 24/7, even when logged off.Automatically

Installing

  1. Double-click Carrier-DependenSee-1.3.0.0-x64.msi.
  2. Approve the User Account Control prompt (Yes). Admin rights let the collector see which program owns each connection.
  3. Complete the wizard and click Install. It installs to C:\Program Files\Carrier DependenSee.
  4. The collector service starts on its own and begins recording immediately.
Nothing else to install. The .NET runtime is bundled inside the app.

First run

Open Carrier DependenSee from the Start Menu. The window has three tabs: Dashboard, History, and Settings. Give the collector a few seconds to complete its first sweep — the Dashboard fills in automatically.

Dashboard tab

Your live, at-a-glance view, refreshing every few seconds. The summary bar shows counts of Services, Listeners, Inbound, and Outbound, followed by three tables: registered services, listening endpoints, and active connections (each connection tagged Inbound or Outbound and by scope). Every connection now shows the owning Service (resolved from the process’s PID) alongside the raw process name. Click any column header to sort. Tick Hide standard Windows services to drop built-in OS services and see only third-party / application services — the ones that actually matter for a migration.

History tab

The Dashboard shows now; History searches the past. Filters apply automatically as you change them (no button needed). Use Export CSV / Export JSON for the raw data, or Export firewall PDF for a migration-ready report (see below). CSV/JSON exports now include a source (machine) column and the resolved service_name, so data from several machines can be combined and traced back. All exports prompt you for a save location.

The Service column shows the actual owning Windows service name instead of a bare executable (e.g. W3SVC rather than svchost), captured at collection time and carried through every grid, export, and the firewall PDF.

Column filters (Excel-style)

Every key column in the History grids — Proto, Direction, Scope, Local/Remote address, ports, Service, and Process — has a small ▼ funnel in its header. Click it to open a checklist of the distinct values actually present in the current results, with a search box and Select all / Clear buttons. Untick the values you don’t want and the grid filters instantly — no re-query. An active funnel turns blue. Column filters stack with each other and with the filter bar above, and your selections are preserved when you re-run the query.

Example: click the Service funnel, clear everything, tick just W3SVC and MSSQLSERVER, and the grid shows only those two services’ traffic.

Unique flows

The collector samples every few seconds, so one connection appears hundreds of times. Tick Unique flows only to collapse those into distinct dependencies. The grid switches to show First seen, Last seen, and Count per flow — so "web server → database on 1433" appears once, with how long and how often it was seen.

Why it's smart about ports: ephemeral client ports are ignored when collapsing, so a busy outbound destination doesn't explode into thousands of "unique" rows. Outbound flows are keyed on the destination; inbound on the service port + source.

Firewall PDF report

This is the deliverable for your network/security team. Click Export firewall PDF to produce a printable report that turns observed traffic into the firewall rules the machine will need after it moves to AWS, Azure, or GCP.

What's in it

  • Machine identity & window — hostname, its IP addresses, the observation period, and the filters used.
  • Summary — counts of inbound rules, outbound rules, and internet-facing flows (the ones security scrutinizes most).
  • Inbound rules — for each service port: the protocol, port, owning program, and the allowed sources (each tagged Private or Internet). Open ports with no observed traffic are flagged.
  • Outbound rules — each destination the machine reached: protocol, IP, port, owning service/program, scope, and (optionally) a resolved hostname.
  • Notes & caveats — reminders about observation windows and IP-based egress.

Options

OptionEffect
Resolve namesBest-effort reverse-DNS so destinations show a hostname alongside the IP. On by default; adds a short lookup step.
Summarize /24Collapse many inbound sources in the same /24 into one subnet entry for cleaner rules. Off by default (listing exact sources is safer).
Tip for accuracy: the report reflects only traffic observed in the window. Let the collector run across a representative period (including nightly jobs and month-end peaks) before you finalize rules. The current filters (time window, family, scope) are applied to the report.

Annotations

Tag any process, port, or host with a friendly name, an owner, a criticality, and notes. On the Annotations tab pick the kind, enter the key (e.g. process sqlservr, port 1433, or a host IP), fill in the details, and click Save / Update. These labels automatically enrich the firewall PDF so reports read in business terms — "1433 (SQL Server) — Finance DB · owner: J. Lee · high criticality" instead of a bare port number. Ports also show their well-known service name (443 → HTTPS, 3389 → RDP, …) automatically.

Dependency map

The Map tab draws the dependencies as a node graph — the machine sits at the center with its peers around it — node color shows scope (gray = private, red = internet), edge color shows direction (blue = outbound, green = inbound), and peer size reflects traffic volume. Adjust the time window and max peers, then click Refresh map.

Remote collection (agentless)

The Remote tab collects services and connections from other servers without installing anything on them. Windows targets are read over WinRM / PowerShell Remoting; Linux targets over SSH. Enter one or more hosts — single names/IPs, dashed ranges (10.0.0.20-40), or CIDR blocks (10.1.0.0/24) — pick the OS/transport (or leave it on Auto), supply a username and password (or an SSH private key), and click Scan. Progress streams into the log, and each host that responds is saved as its own database and added to the Fleet view automatically. Continuous collection: a remote scan is a point-in-time snapshot, but you can schedule it. Enter a profile name and an interval, then click Schedule — Carrier DependenSee registers a Windows Scheduled Task that re-runs the scan on that cadence (even when the viewer is closed), building history in each host's database. Use an SSH key for unattended Linux runs so no password is stored; for password profiles the password is saved DPAPI-encrypted for your Windows account. Click Unschedule to remove the task.

Prerequisites & safety: only scan systems you are authorized to access. Windows targets need WinRM enabled (Enable-PSRemoting; for workgroup hosts add them to the client's TrustedHosts). Linux targets need SSH reachable. Privileged credentials (local admin / WinRM, or root / sudo) are needed for full process-to-service attribution. Credentials are used only for the scan and are not saved.

Fleet view

The Fleet tab shows every machine you've collected — whether imported from a servicemap.db file or gathered on the Remote tab — unioned together. It lists each machine's unique flows and, most usefully for migration planning, the machine-to-machine dependencies: which servers talk to which. Assign each machine a wave and the view flags dependencies that cross a wave boundary, so you can group chatty systems into the same migration wave. Select a machine in the list to narrow the flows and dependencies to just that server, and click Export server PDF to generate a firewall-rule report scoped to that one machine — the same migration-ready document as the History tab.

Viewing any server (active machine)

The Machine dropdown in the top-right switches which server the Dashboard, History, Map, and firewall PDF are showing. Pick Local (this machine) for the live collector, or any fleet/remote machine to explore its snapshot with the full toolset — exactly as you would the local server. From the Fleet tab you can also select a machine and click View this server to jump straight to it. A banner appears while a remote snapshot is active (service controls are disabled, since they only apply to the local collector); switch back to Local to return.

Firewall reconciliation

The Firewall tab compares what each server actually connects to against your firewall policy. Point Policy folder at a folder of exported CSVs (Palo Alto egress/ingress rules, a Check Point rule export, and the address-group objects) and click Load policy. Then choose a server and click Reconcile. For every observed flow it:

  • Resolves the remote IP to its firewall object and group membership (e.g. 10.94.9.167 → cmusswhadpdc01).
  • Classifies coverage: Covered (an existing allow rule permits it, and names the rule), Gap (real traffic with no rule — a firewall change needed for migration), or Denied (matched by a specific block rule).

Filter by coverage (e.g. show only Gap to get the net-new rules list) and Export CSV for your network team. The firewall files are read as-is and indexed in memory; re-load whenever you receive fresh exports.

Notes: matching is port-based (Palo Alto app-id is advisory); Check Point objects can only be resolved where their IP is known, so on-prem coverage improves once an address-object export is available. NAT boundaries can make observed IPs differ from rule IPs.

Settings tab

Shows the collector service status with Install / Start / Stop / Uninstall buttons (each prompts for elevation). Below that you can set the Database path, Export directory, and Dashboard refresh interval — click Apply settings after changes. Normally the MSI has already installed and started the service, so you won't need these buttons.

Common tasks

History → Scope = Internet, Unique flows only, Run query. Then Export firewall PDF for the write-up.

Produce the migration firewall rules for this machine

History → set Hours back to cover a representative period, leave Family = IPv4, click Export firewall PDF.

Find who connects to a service port

History → Local port = e.g. 1433, Direction = Inbound, Unique flows only, Run query.

Managing the collector service

It's a normal Windows Service named Carrier DependenSee Collector. Control it from the Settings tab, or with Win+R → services.msc. Stopping it pauses recording; it restarts automatically with Windows.

Uninstalling

  1. Settings → Apps → Installed apps (or Control Panel → Programs and Features).
  2. Find Carrier DependenSee and choose Uninstall.
  3. The collector service is stopped and removed automatically.
Recorded data under C:\ProgramData\CarrierDependenSee is left in place; delete that folder to remove the history too.

FAQ

Does it send my data anywhere?

No. Everything stays on the machine in a local database. The only network activity is optional reverse-DNS lookups when you generate a firewall PDF with "Resolve names" enabled.

Does it slow the machine down?

No — a lightweight snapshot every few seconds, written locally.

Can it run on a server with no one logged in?

Yes. The collector runs as a system service and records continuously.

What about IPv6?

IPv6 is always collected. The Family filter lets you focus on IPv4 (the usual migration case) or IPv6 as needed.

What's new in 1.7–1.9

  • ETW event capture — short-lived connections between polling sweeps are now recorded (requires the collector to run elevated).
  • Flow aggregation — distinct dependencies are stored compactly; raw samples keep 7 days, flows keep 30 (configurable).
  • Remote scan bursts & conntrack — scheduled agentless scans take several snapshots per session and harvest recently-closed flows from Linux connection tracking.
  • Firewall reconciliation — protocol-aware matching, multi-file policy loading, rule zones in the grid, and Find unused rules (allow rules never exercised in the window).
  • Coverage honesty — the firewall PDF states how many sweeps its data represents and how it was collected.
  • Fixes — History/flow timestamps now display correctly; direction inference survives listener restarts.
